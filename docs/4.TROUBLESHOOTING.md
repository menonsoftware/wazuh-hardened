# Troubleshooting Guide

Use this guide to diagnose common deployment and runtime issues.

## 1. `start_all.sh` Fails Immediately

Symptom:

- Script exits with `vm.max_map_count` error.

Cause:

- Host kernel parameter too low for OpenSearch.

Fix:

```bash
sudo sysctl -w vm.max_map_count=262144
echo "vm.max_map_count=262144" | sudo tee -a /etc/sysctl.conf
```

Verify:

```bash
cat /proc/sys/vm/max_map_count
```

## 2. Indexer/Searchhead Security Not Initializing

Symptoms:

- Start sequence stalls during security admin step
- Dashboard cannot authenticate to indexer

Likely causes:

- Certificate DN mismatch
- Incorrect indexer credentials in `.env`

Checks:

```bash
openssl x509 -noout -subject -in certs/admin.pem
```

- Ensure `O=` in certificate subject matches root `.env` `ORGANIZATION`.
- Ensure OpenSearch DN entries in:
  - `regional/opensearch.yml`
  - `central/opensearch.yml`
  match generated certificate subjects.

Recovery sequence:

1. Fix `.env` certificate subject values.
2. Regenerate certificates:

```bash
./scripts/ssl.sh
```

3. Remove security sentinel for affected component only:

```bash
rm -f regional/state/.security_initialized
rm -f central/state/.security_initialized
```

4. Start again:

```bash
./scripts/start_all.sh
```

## 3. Dashboard Not Reachable Through Traefik

Symptoms:

- `https://<DASHBOARD_FQDN>` times out or returns routing error.

Checks:

1. DNS resolves to host:

```bash
getent hosts <DASHBOARD_FQDN>
```

2. Edge network exists:

```bash
docker network inspect edge-net >/dev/null && echo OK
```

3. Edge and central services are running:

```bash
cd edge && docker compose ps
cd ../central && docker compose ps
```

4. Review Traefik logs:

```bash
cd edge && docker compose logs -f traefik
```

## 4. Helpdesk (Zammad) Not Reachable

Symptoms:

- `https://<HELPDESK_FQDN>` fails
- 502/504 or TLS errors

Checks:

```bash
cd zammad && docker compose ps
cd zammad && docker compose logs -f zammad-nginx
cd zammad && docker compose logs -f zammad-railsserver
```

Also confirm DNS for `HELPDESK_FQDN` points to edge host.

## 5. Permission Denied on Data Mounts

Symptoms:

- Services crash on startup with file permission errors in data directories.

Cause:

- Host ownership drift for data paths mounted into containers.

Fix:

```bash
./scripts/ssl.sh
```

Or apply ownership manually:

```bash
docker run --rm -v "$(pwd):/soc" alpine sh -c "chown -R 1000:1000 /soc/regional/data /soc/central/data /soc/zammad/data"
```

## 6. Security Initialization Was Skipped Unexpectedly

Symptom:

- You changed credentials/certs but script reports security already initialized.

Cause:

- Sentinel files still present.

Fix:

```bash
rm -f regional/state/.security_initialized
rm -f central/state/.security_initialized
./scripts/start_all.sh
```

Only remove sentinels when re-initialization is intentional.

## 7. OpenSearch Cluster Health is Red/Yellow

Checks:

```bash
cd regional && docker compose logs -f wazuh.indexer.1
cd regional && docker compose logs -f wazuh.indexer.2
```

```bash
curl -k -u "<INDEXER_USER>:<INDEXER_PASSWORD>" https://localhost:9200/_cluster/health
```

Possible causes:

- Insufficient memory/CPU limits
- Storage pressure
- Node connectivity problems

Actions:

- Increase indexer memory limits and JVM heap in `regional/.env`
- Verify both indexers are healthy and discoverable
- Restart affected component after configuration changes

## 8. ACME Certificate Issues in Traefik

Symptoms:

- Browser sees invalid cert
- Traefik cannot obtain Let's Encrypt certificates

Checks:

- `TRAEFIK_ACME_EMAIL` is set in `edge/.env`
- Inbound ports `80` and `443` are open
- FQDN resolves publicly to the host

Logs:

```bash
cd edge && docker compose logs -f traefik
```

## 9. Backup Completed but Data Missing

Likely cause:

- `scripts/backup.sh` does not include all component data by default (for example, Zammad data).

Action:

- Extend backup procedure manually or update backup script to include additional paths.
- Test restore process periodically to validate archive completeness.

## 10. Last-Resort Clean Rebuild

If multiple issues persist and rapid recovery is preferred:

```bash
./scripts/erase.sh
./scripts/ssl.sh
./scripts/start_all.sh
```

Warning:

- `erase.sh` is destructive and removes persistent data.

## 11. Wazuh Agent Fails to Start (`Invalid server address: MANAGER_IP`)

Symptoms:

- Agent service fails during startup.
- Logs show:
  - `wazuh-agentd: ERROR: (4112): Invalid server address found: 'MANAGER_IP'`
  - `wazuh-agentd: ERROR: (1215): No client configured. Exiting.`

Cause:

- The agent configuration still contains the literal placeholder `MANAGER_IP` instead of a real manager IP/FQDN.
- Commonly caused by running install commands copied from examples without replacing the variable.

Fix:

1. Update the agent manager address in `ossec.conf`:

```bash
sudo nano /var/ossec/etc/ossec.conf
```

Set a valid manager endpoint, for example:

```xml
<client>
  <server>
    <address>10.0.0.91</address>
    <port>1514</port>
    <protocol>tcp</protocol>
  </server>
</client>
```

2. Or replace the placeholder directly:

```bash
sudo sed -i "s|MANAGER_IP|10.0.0.91|g" /var/ossec/etc/ossec.conf
```

3. Restart the agent:

```bash
sudo systemctl restart wazuh-agent
sudo systemctl status wazuh-agent --no-pager -l
```

Verify:

```bash
sudo grep -R "MANAGER_IP" /var/ossec/etc || true
sudo tail -n 80 /var/ossec/logs/ossec.log
```

Optional reinstall pattern (correct variable usage):

```bash
sudo WAZUH_MANAGER='10.0.0.250' WAZUH_AGENT_NAME="$(hostname -s)" bash ./wazuh-install.sh -a
```
