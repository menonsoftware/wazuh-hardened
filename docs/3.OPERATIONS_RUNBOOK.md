# Operations Runbook

This runbook covers day-2 operations for the SOC demo environment.

## 1. Start and Stop

Start all components in orchestrated order:

```bash
./scripts/start_all.sh
```

Stop and remove all components in reverse order:

```bash
./scripts/stop_all.sh
```

Component order in stop script:

- `edge`
- `zammad`
- `central`
- `regional`

## 2. Build and Rebuild

Build images for all components:

```bash
./scripts/build_all.sh
```

Build a single component manually:

```bash
cd regional && docker compose build
```

Use this pattern for `central`, `zammad`, or `edge` as needed.

## 3. Health and Status Checks

Check container states:

```bash
cd regional && docker compose ps
cd ../central && docker compose ps
cd ../zammad && docker compose ps
cd ../edge && docker compose ps
```

Follow logs:

```bash
cd regional && docker compose logs -f
```

Repeat for other components.

Quick ingress checks:

```bash
curl -kI https://<DASHBOARD_FQDN>
curl -kI https://<HELPDESK_FQDN>
```

## 4. Component Lifecycle Commands

Start or restart a single component:

```bash
cd <component>
docker compose up -d
```

Stop a single component:

```bash
cd <component>
docker compose down
```

Where `<component>` is one of:

- `regional`
- `central`
- `zammad`
- `edge`

## 5. Backup Procedure (Cold Backup)

Use built-in backup script:

```bash
./scripts/backup.sh
```

Behavior:

1. Stops `regional`, `central`, and `edge`
2. Creates archive under `backup/` named `wazuh_backup_<timestamp>.tar.gz`
3. Restarts stopped services

Backup includes:

- `central/data`
- `regional/data`
- `edge/data`
- `certs`
- `.env` files for root, `central`, `regional`, `edge`

Note:

- Zammad data is not part of `backup.sh` by default.

## 6. Restore Procedure (Manual)

There is no dedicated restore script. Use this manual sequence.

1. Stop all services.

```bash
./scripts/stop_all.sh
```

2. Extract backup archive to project root.

```bash
tar -xzf backup/wazuh_backup_<timestamp>.tar.gz -C .
```

3. Reapply ownership if required.

```bash
docker run --rm -v "$(pwd):/soc" alpine sh -c "chown -R 1000:1000 /soc/regional/data /soc/central/data /soc/zammad/data"
```

4. Start stack.

```bash
./scripts/start_all.sh
```

5. Validate with `docker compose ps` and endpoint checks.

## 7. Full Reset (Destructive)

Run only when you want a clean rebuild from scratch.

```bash
./scripts/erase.sh
```

This removes:

- Containers and volumes in all components
- `certs/`
- Persistent `data/` directories
- `state/` directories and security sentinels

After reset, redeploy from the deployment guide starting at certificate generation.

## 8. Security Initialization Sentinel Behavior

First-run security initialization is tracked by:

- `regional/state/.security_initialized`
- `central/state/.security_initialized`

If these files exist, `start_all.sh` skips security re-initialization.

Use with care:

- Remove a sentinel only when you intentionally want to re-run security bootstrap for that component.

## 9. Scaling and Capacity Knobs

Primary scaling controls are in `.env` files:

- `INDEXER_JAVA_OPTS`
- `INDEXER_MEMORY_LIMIT`, `INDEXER_CPU_LIMIT`
- `MANAGER_MEMORY_LIMIT`, `MANAGER_CPU_LIMIT`
- `SEARCHHEAD_*`, `DASHBOARD_*`

For larger deployments:

- Increase indexer node count in `regional/docker-compose.yml`
- Update seed hosts and node definitions in `regional/opensearch.yml`
- Scale manager topology with additional manager workers/clusters

## 10. Routine Maintenance Checklist

- Check all components are healthy (`docker compose ps`)
- Review container logs for repeated errors
- Confirm certificate validity windows and rotation plan
- Verify backups are created and restorable
- Rotate application credentials and API passwords on schedule
